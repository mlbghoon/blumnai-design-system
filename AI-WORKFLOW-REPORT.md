# AI 기반 개발 워크플로우 보고서

## Executive Summary

| 항목 | 결과 |
|------|------|
| **개발 속도** | 컴포넌트당 개발 시간 약 60-70% 단축 |
| **디자인 정확도** | Figma 스펙 직접 연동으로 수동 값 입력 오류 제거 |
| **품질 보증** | TypeScript 검증 + 접근성 체크리스트 자동 실행 |
| **지식 보존** | Figma 스펙, 컴포넌트 패턴이 코드베이스에 문서화 |

**핵심 성과:**
- AI가 코드 작성을 담당하고, 개발자는 시각적/기능적 검증에 집중하는 역할 분담 확립
- Figma MCP + REST API 이중 연동으로 디자인-개발 간 불일치 최소화
- 프로젝트별 Claude Code Skills 설정으로 일관된 코드 품질 자동 유지

---

## 개요
본 보고서는 디자인 시스템 컴포넌트 개발 시 AI 어시스턴트(Claude)와의 협업 워크플로우 패턴을 문서화합니다.

## 워크플로우 방법론

### 1단계: 기획 및 명세 정의
- **AI 플랜 모드**를 활용하여 코드 작성 전 요구사항 분석
- AI가 코드베이스 탐색, Figma 스펙 확인 후 구현 계획 수립
- 계획 검토 및 승인 후 실행 착수
- **장점**: 잘못된 방향으로의 개발 및 범위 확장 방지

### 2단계: 시각적 피드백 기반 반복 구현

```
┌─────────────────────────────────────────────────────────┐
│  개발자가 스펙/계획 제공                                   │
│              ↓                                          │
│  AI가 코드 변경 구현                                      │
│              ↓                                          │
│  개발자가 렌더링 결과 확인 (브라우저/Storybook)             │
│              ↓                                          │
│  개발자가 스크린샷 + 간단한 피드백 제공                     │
│              ↓                                          │
│  AI가 구현 수정                                          │
│              ↓                                          │
│  시각적으로 정확할 때까지 반복                             │
└─────────────────────────────────────────────────────────┘
```

**주요 특징:**
- **스크린샷**과 간결하고 명확한 피드백 전달 (장문의 설명 불필요)
- **간결한** 수정 지시 ("padding is not 4px", "only border-bottom is needed")
- AI가 시각적 차이를 해석하고 자율적으로 수정안 제시

### 3단계: 기능 및 인터랙션 검증

시각적 검증 외에도 다음 항목들을 테스트하여 피드백 제공:

**키보드 네비게이션**
- Tab 키로 포커스 이동이 논리적 순서로 되는지 확인
- Enter/Space 키로 선택 및 활성화 동작 확인
- 방향키(Arrow keys)로 옵션 간 이동 확인 (예: 캘린더 날짜 이동, 드롭다운 항목 이동)
- Escape 키로 모달/팝오버 닫기 동작 확인

**포커스 상태**
- 포커스 링(focus ring)이 디자인 스펙대로 표시되는지 확인
- 포커스 트랩이 필요한 컴포넌트(모달, 다이얼로그)에서 제대로 작동하는지 확인

**상태 변화**
- hover, active, disabled 상태가 올바르게 표시되는지 확인
- 로딩 상태 전환이 자연스러운지 확인
- 에러 상태 표시 및 해제가 정상 동작하는지 확인

**이벤트 핸들러**
- onClick, onChange 등 콜백이 올바른 값과 함께 호출되는지 확인
- 폼 제출 시 validation이 정상 작동하는지 확인
- controlled/uncontrolled 모드 모두 정상 동작하는지 확인

**엣지 케이스**
- 긴 텍스트 입력 시 truncation 또는 줄바꿈 처리 확인
- 빈 상태(empty state) 표시 확인
- 다크 모드 전환 시 모든 색상이 올바르게 변경되는지 확인

**접근성**
- aria-label, aria-describedby 등 ARIA 속성이 올바르게 설정되는지 확인

**피드백 방식:**
- 문제 발견 시 간단히 설명 ("can not fucust on component using tab", "no action firing on Enter key")
- 필요시 브라우저 DevTools 콘솔 스크린샷 공유
- AI가 해당 이벤트 핸들러 또는 접근성 속성 수정

### 4단계: Figma 연동 및 리소스 통합

**두 가지 Figma 연동 방식 활용:**

| 방식 | 용도 | 장점 |
|------|------|------|
| **Figma MCP** | AI가 실시간으로 Figma 파일 직접 접근 | 별도 설정 없이 즉시 조회 가능, 대화 중 바로 스펙 확인 |
| **Figma REST API** | 스크립트를 통한 상세 데이터 조회 및 저장 | JSON 형태로 코드베이스에 보존, 팀 공유 용이 |

**활용 흐름:**
```
┌─────────────────────────────────────────────────────────┐
│  개발자가 Figma URL 또는 노드 ID 제공                     │
│              ↓                                          │
│  AI가 Figma MCP로 빠르게 스펙 확인                        │
│              ↓                                          │
│  상세 데이터 필요시 REST API로 조회                       │
│              ↓                                          │
│  디자인 데이터를 /source/figma-*.json에 저장              │
│              ↓                                          │
│  저장된 스펙 기반으로 구현 진행                           │
└─────────────────────────────────────────────────────────┘
```

**장점:**
- **단일 진실 공급원(Single Source of Truth)**: Figma 스펙이 코드와 함께 버전 관리됨
- **재사용성**: 저장된 스펙을 향후 수정 작업 시 참조
- **팀 협업**: 다른 개발자도 동일한 스펙 파일 활용 가능

## 커뮤니케이션 패턴

### 시각적 피드백

| 개발자 입력 | AI 응답 |
|-------------|---------|
| 스크린샷 + "padding is not 8px" | Figma 스펙 재확인 후 수정 |
| 스크린샷 + "only border-bottom is needed" | 해당 스타일 수정 |
| Figma 스크린샷 + 개발된 컴포넌트 스크린샷 비교 | 차이점 식별 후 수정안 제시 |

### 기능 피드백

| 개발자 입력 | AI 응답 |
|-------------|---------|
| "can't focus with Tab" | tabIndex 또는 키보드 이벤트 핸들러 수정 |
| "Enter key not working" | onKeyDown 핸들러 추가/수정 |
| "focus ring not visible" | focus-visible 스타일 수정 |
| 콘솔 에러 스크린샷 | 에러 원인 분석 및 수정 |

### Figma 연동 및 Skills 활용

| 개발자 입력 | AI 응답 (자동) |
|-------------|----------------|
| Figma URL 공유 | Figma MCP로 스펙 확인 |
| "save this" / "save specs" | `figma-save` 스킬로 `/source/` 폴더에 JSON 저장 |
| "create new component" | `new-component` 스킬로 파일 구조 생성 |
| Story 파일 작업 시작 | `storybook-story` 스킬로 컨벤션 자동 참조 |
| 구현 완료 | `component-checklist` 스킬로 검증 자동 실행 |

### 방향 제시

| 개발자 입력 | AI 응답 |
|-------------|---------|
| "we might need dedicated component" | 기존 컴포넌트 대신 새 컴포넌트 생성 |
| "don't use Button, make custom one" | 의존성 없는 커스텀 구현 |
| "let's try this approach" | 제안된 방향으로 전략 수정 |

**효율성 향상**: 최소한의 컨텍스트 전환. 개발자는 브라우저/Figma에서 결과물 검토에 집중, AI가 코드 변경 처리.

개발자가 각 컴포넌트에 대해 수행하는 검증 항목:

| 카테고리 | 검증 항목 | 확인 방법 |
|----------|-----------|-----------|

| 시각적 | 반응형 | 뷰포트 크기 조절 |

| 키보드 | 선택/활성화 | Enter/Space 키 |
| 키보드 | 취소/닫기 | Escape 키 |

## 제공 가치

### 1. **속도**
- 빠른 반복 주기 (피드백 → 수정 → 검증이 수 분 내 완료)
- AI가 보일러플레이트 처리, 개발자는 시각적 검증에 집중

### 2. **정확도**
- Figma MCP + REST API 연동으로 픽셀 단위 정확한 스펙 확보
- 디자인 값 수동 옮겨적기 불필요

### 3. **품질**
- 시각적 검증뿐 아니라 키보드 접근성, 상태 관리까지 체계적 검증
- WCAG 가이드라인 준수 여부 확인

### 4. **문서화**
- Figma 스펙 자동으로 코드베이스에 저장
- 명확한 근거와 함께 코드 변경 추적

### 5. **컨텍스트 전환 감소**
- 개발자는 브라우저에서 검토
- AI가 파일 탐색, 코드 편집, TypeScript 검사 처리
- 관련 파일 수동 검색 불필요

### 6. **지식 축적**
- 디자인 토큰, 컴포넌트 패턴, Figma 데이터 보존
- 향후 개발자(또는 AI)가 저장된 스펙 참조 가능

## 효율성 비교

### 기존 방식 vs AI 협업 방식

| 작업 단계 | 기존 방식 | AI 협업 방식 | 개선 효과 |
|-----------|-----------|--------------|-----------|
| Figma 스펙 확인 | 수동으로 값 확인 및 기록 | AI가 Figma API로 직접 조회 | 오류 제거, 시간 단축 |
| 보일러플레이트 작성 | 파일 구조, 타입, 스토리 수동 생성 | Skills로 자동 생성 | 반복 작업 제거 |
| 스타일 구현 | Figma 값을 CSS로 수동 변환 | AI가 디자인 토큰 기반 자동 적용 | 일관성 확보 |
| 접근성 검증 | 별도 테스트 단계 필요 | 구현 중 체크리스트 자동 실행 | 누락 방지 |

### 반복 주기 개선

```
기존 방식:
코드 작성 → 브라우저 확인 → 문제 파악 → 파일 탐색 → 수정 → 반복
(1회 주기: 10-15분)

AI 협업 방식:
스크린샷 전달 → AI 수정 → 확인
(1회 주기: 2-3분)
```

## 리스크 감소

### 1. 디자인-개발 불일치 방지

| 리스크 | 기존 방식 | AI 협업 방식 |
|--------|-----------|--------------|
| Figma 값 오기입 | 수동 확인으로 휴먼 에러 발생 가능 | Figma API 직접 조회로 원천 차단 |

### 2. 코드 품질 보장

| 검증 항목 | 자동화 방식 |
|-----------|-------------|
| TypeScript 타입 오류 | AI가 `tsc --noEmit` 실행 후 수정 |
| 디자인 시스템 규칙 위반 | `design-system-rules` 스킬이 비표준 클래스 감지 |
| 접근성 누락 | `component-checklist` 스킬이 aria 속성 확인 |
| Storybook 컨벤션 | `storybook-story` 스킬이 패턴 강제 |

### 3. 지식 손실 방지

| 리스크 | 해결 방식 |
|--------|-----------|
| 담당자 부재 시 스펙 확인 불가 | Figma 스펙이 `/source/` 폴더에 JSON으로 보존 |
| 구현 의도 불명확 | 코드와 함께 디자인 근거 문서화 |

### 4. 일정 리스크 감소

- **예측 가능한 개발 속도**: 반복 주기가 짧아 진행 상황 파악 용이
- **블로커 최소화**: AI가 파일 탐색, 패턴 파악을 즉시 처리
- **재작업 감소**: 플랜 모드로 방향 합의 후 구현 착수

## Claude Code Skills

프로젝트별 맞춤 Skills를 설정하여 AI의 일관성과 효율성을 높입니다. Skills는 `.claude/skills/` 디렉토리에 Markdown 파일로 정의됩니다.

### 설정된 Skills

모든 Skills는 **자동 호출**되도록 설정됨. AI가 컨텍스트에 따라 적절한 시점에 자동으로 실행.

| Skill | 자동 호출 시점 | 용도 |
|-------|---------------|------|
| `design-system-rules` | 스타일링 코드 작성 시 | 디자인 시스템 CSS 규칙 자동 참조 |
| `figma-save` | Figma URL 공유 시 | Figma 스펙 조회 및 저장 |
| `component-checklist` | 구현 완료 직전 | 검증 체크리스트 자동 실행 |
| `storybook-story` | Story 파일 생성/수정 시 | Storybook 컨벤션 가이드 |
| `new-component` | 새 컴포넌트 요청 시 | 파일 구조 및 보일러플레이트 생성 |

### 자동 호출 예시

```
User: "create Calendar component"
→ AI가 new-component 스킬 자동 실행

User: Figma URL 공유 + "save this"
→ AI가 figma-save 스킬 자동 실행

AI: 구현 완료 후 사용자에게 보고 전
→ AI가 component-checklist 스킬 자동 실행

User: "add story"
→ AI가 storybook-story 스킬 자동 실행
```

### 자동화의 장점

- **일관성**: 매번 동일한 규칙과 패턴 적용
- **누락 방지**: 체크리스트 자동 실행으로 검증 단계 생략 불가
- **효율성**: 개발자가 명시적으로 요청하지 않아도 필요한 시점에 실행
- **컨텍스트 인식**: AI가 대화 맥락을 파악하여 적절한 스킬 선택

### Skills의 장점

1. **일관성**: 디자인 시스템 규칙을 AI가 항상 준수
2. **자동화**: 반복 작업(Figma 저장, 파일 생성) 단순화
3. **품질 보증**: 체크리스트로 누락 방지

### 파일 위치

```
.claude/skills/
├── design-system-rules/SKILL.md   # CSS 규칙 (자동 참조)
├── figma-save/SKILL.md            # Figma 저장 자동화
├── component-checklist/SKILL.md   # 검증 체크리스트
├── storybook-story/SKILL.md       # Story 컨벤션
└── new-component/SKILL.md         # 컴포넌트 보일러플레이트
```

## 팀 도입 권장사항

1. **플랜 모드 활용** — 단순하지 않은 기능은 계획 수립 후 진행하여 불필요한 반복 감소
2. **시각적 커뮤니케이션** — 스크린샷이 설명보다 빠름
3. **Figma 연동 설정** — MCP로 실시간 접근, REST API로 스펙 저장
4. **디자인 스펙 저장** — Figma에서 조회 후 코드베이스에 보존
5. **반복 루프 신뢰** — AI가 시도하게 하고, 결과물 검증 후 간단히 수정 지시
6. **피드백은 간결하고 정확하게** — AI가 이전 대화 컨텍스트에서 맥락 파악
7. **기능 검증 포함** — 시각적 확인 후 키보드, 접근성, 상태 변화도 테스트
8. **Skills 설정** — 프로젝트별 규칙과 자동화를 `.claude/skills/`에 정의하여 AI 일관성 확보

이 워크플로우는 AI를 **구현을 담당하는 페어 프로그래머**로, 개발자를 **시각적 QA 및 프로덕트 오너**로 활용합니다. **Figma MCP와 REST API의 이중 연동**을 통해 디자인 스펙을 정확하게 확보하고, 개발자의 전문성은 코드 타이핑이 아닌 **시각적 검증, 인터랙션 테스트, 접근성 확인** 등 판단과 검증에 집중되어, 품질 기준을 유지하면서 더 빠른 반복이 가능합니다.
