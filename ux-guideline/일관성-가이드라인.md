# Deskit UI 제작 AI 규칙서

이 문서는 AI가 UI를 생성·수정할 때 따라야 하는 실행 규칙이다.
설명보다 적용 우선이다.
모호할 때는 이 문서의 우선순위와 결정 순서를 따른다.
일관성-지식베이스.md에서 세부 디자인 요소를 참고한다.

---

## 0. 우선순위

항상 아래 우선순위로 판단한다.

1. DS 컴포넌트 사용 가능 여부
2. 앱 레이아웃 전용 토큰(`bg-app-*`, `text-app-*`, `border-app-*`)
3. DS 시맨틱 토큰 (`bg-default`, `text-subtle`, `border-default` 등)
4. 기존 파일/주변 코드 패턴 유지
5. 특수 케이스에 한해 하드코딩 허용 (브랜드 컬러 적용, 채팅 플래그 색상 맵, 채팅 말풍선별 개별 색상, gradient 배경, 시맨틱 토큰에 없는 유색 아이콘/배경)

---

## 1. 절대 규칙

### 1.1 컴포넌트
- 버튼, 입력, 선택, 체크박스, 라디오, 스위치, 탭, 다이얼로그, 툴팁, 팝오버, 아바타, 뱃지, 스크롤 영역, 토스트, 날짜 선택, 데이터 그리드는 직접 구현하지 말고 DS 컴포넌트를 우선 사용한다.
- DS 컴포넌트가 있으면 자체 구현보다 DS를 사용한다.
- DS 컴포넌트 스타일 오버라이드는 최소화한다.
- 스타일 변경은 `variant`, `size`, `shape`, `className` 같은 공식 prop을 우선 사용한다.

### 1.2 색상
- `bg-white`, `bg-black`, `bg-gray-*` 사용 금지
- `text-gray-*` 사용 금지
- `border-gray-*` 사용 금지
- `bg-[var(--app-*)]` arbitrary value 사용 금지
- 레이아웃 색상은 `bg-app-*`, `text-app-*`, `border-app-*` 사용
- 일반 surface는 DS 배경 토큰 사용
- gray 계열 hex 하드코딩 금지

### 1.3 타이포그래피
- `text-xs`, `text-sm`, `text-base`, `text-lg` 등 Tailwind 텍스트 크기 사용 금지
- 텍스트 크기는 `size-*`만 사용
- `font-light`, `font-bold`, `font-extrabold` 사용 금지
- 굵기는 `font-normal`, `font-medium`, `font-semibold`만 사용

### 1.4 인터랙션
- `transition-all` 금지
- 변경되는 속성만 명시한 transition 사용
- 인터랙티브 요소는 최소 40x40px 히트 영역 확보
- 아이콘 전용 버튼에는 툴팁 적용

### 1.5 라운딩
- 임의 radius 값 금지
- 같은 역할의 요소는 같은 radius 사용
- 기본 3단계:
  - 말단 요소: `rounded-sm`
  - 컨테이너: `rounded-md`
  - 콘텐츠 블록/모달: `rounded-lg`

### 1.6 Primary 색상 사용
- Primary 색상은 시각적으로 매우 두드러지므로 남용 금지
- 버튼(`style="primary"`), 포인트 컬러, 강조 아이콘/뱃지 등 primary 계열 색상을 사용하는 모든 요소에 적용되는 규칙이다
- **1개 컨테이너(카드, 패널 등)당 primary 색상 요소는 최대 1개**만 사용한다
- 모든 컨테이너에 primary를 넣을 필요는 없다 — 가장 중요한 CTA나 핵심 강조 요소에만 사용한다
- **전체 화면 기준 primary 색상 요소는 5개를 넘지 않는다**
- 일반적인 기능 버튼에는 `style="secondary"` 또는 `style="soft"`를 사용한다
- primary를 배치할 때는 "이 화면에서 사용자의 시선이 가장 먼저 가야 하는 요소가 무엇인가"를 기준으로 판단한다

## 2. UI 제작 결정 트리

### 2.1 새 UI를 만들 때
1. DS 컴포넌트로 표현 가능한지 확인한다.
2. 레이아웃 영역인지 일반 surface인지 판단한다.
3. 색상 토큰을 선택한다.
4. 텍스트 크기와 굵기를 선택한다.
5. spacing/radius/border/shadow를 선택한다.
6. 인터랙션과 툴팁 필요 여부를 판단한다.
7. z-index가 필요한 경우 기존 표 안에서 선택한다.

### 2.2 배경색 결정 순서
1. 앱 레이아웃 영역이면 `bg-app-*`
2. 일반 콘텐츠 배경이면 `bg-default`
3. 약한 구분이 필요하면 `bg-subtle`
4. 더 강한 구분, 토글 OFF, 핸들, placeholder 성격이면 `bg-muted`
5. 오버레이면 `bg-overlay`
6. 반전 강조면 `bg-inverted`
7. disabled 상태면 `bg-state-disabled`
8. 상태/브랜드/말풍선 등 DS로 표현 불가한 특수 케이스만 하드코딩 허용

### 2.3 텍스트 색상 결정 순서
1. 기본 본문: `text-default`
2. 보조 설명: `text-subtle`
3. 캡션/메타: `text-muted`
4. 힌트/placeholder: `text-hint`
5. 시맨틱 상태는 상태 토큰 또는 시맨틱 아이콘 토큰 사용

### 2.4 보더 결정 순서
1. 기본 구분선: `border-default`
2. 더 강한 구분: `border-darker`
3. 가장 강한 구분: `border-strong`
4. 상태 보더: `border-destructive`, `border-success`, `border-warning`, `border-informative`

### 2.5 radius 결정 순서
1. 버튼, 칩, 뱃지, 입력필드: `rounded-sm`
2. 이들을 감싸는 래퍼: `rounded-md`
3. 카드, 말풍선, 패널, 모달: `rounded-lg`
4. 원형/알약형만 `rounded-full`

### 2.6 카드/컨테이너 판단
1. 눈에 띄어야 하면 강조 카드
2. 정보 나열이면 플랫 카드

적용 규칙:
- 강조 카드: `rounded-lg` + `border-default` + `shadow-component-default`
- 플랫 카드: shadow 없이 `bg-default` 또는 `bg-subtle` + padding 중심 구성

### 2.7 리스트 판단
1. 표 형태 데이터면 DS Table/DataGrid 사용
2. 자유 레이아웃 나열이면 Card List
3. Card List는 아래 둘 중 하나만 선택
- Gap 리스트: 아이템 사이 간격으로 구분
- Gapless 리스트: 컨테이너 내부 구분선으로 구분

금지:
- 같은 리스트에서 gap과 `border-b`를 혼용하지 않는다.

## 3. 영역별 규칙

### 3.1 레이아웃
- 전체 컨테이너: `bg-app-container`
- GNB: `bg-app-gnb`, `text-app-gnb-text`, `border-app-gnb-border`
- 사이드바: `bg-app-sidebar`
- 대화방 헤더/본문/컴포저: 각 `bg-app-room-*`, `bg-app-composer`
- 참조 패널: `bg-app-ref-*`
- 모달: `bg-app-modal`
- 검색/공지: `bg-app-search`, `bg-app-notice`

### 3.2 타이포그래피
- 캡션/뱃지/메타: `size-xs`
- 본문/라벨/리스트 아이템: `size-sm`
- 섹션 제목/강조: `size-md`
- 페이지/패널 제목: `size-lg`

### 3.3 폰트 굵기
- 본문: `font-normal`
- 버튼/입력 라벨: `font-medium`
- 섹션 제목/강조: `font-semibold`

### 3.4 간격
- `p-*`, `px-*`, `py-*`는 가능한 DS `padding-*` 계열로 변환한다.
- `gap`, `margin`, `space-*`는 Tailwind 유지 가능
- `panelSpacing.ts` 상수가 있으면 우선 사용한다.

### 3.5 인터랙션
- 리스트 아이템: `hover:bg-subtle transition-colors`
- ghost 버튼: `hover:bg-muted transition-colors`
- 아이콘 버튼: `hover:bg-muted transition-colors`
- 카드 hover: `hover:shadow-sm transition-shadow`
- press feedback: 필요 시 `active:scale-[0.96]`

### 3.6 모달
- 세로 크기는 기본적으로 콘텐츠 높이에 맞춘다
- 뷰포트를 넘기면 `max-h-[calc(100vh-48px)]` + 내부 스크롤
- 특별한 지시가 없으면 중앙 dissolve 방식으로 노출

### 3.7 툴팁
- HTML `title` 사용 금지
- `TooltipTrigger` 사용
- 아이콘 전용 버튼: 툴팁 필수
- 잘린 텍스트: 툴팁 필수
- 단축키 있는 액션: `badge` 사용 가능

## 4. 허용되는 예외

아래는 DS 토큰으로 표현할 수 없는 경우에만 허용한다.

- 상태 플래그 색상 맵
- 브랜드/아이덴티티 컬러
- 채팅 말풍선별 개별 색상
- gradient 배경
- 시맨틱 토큰에 없는 유색 아이콘/배경

단, 회색 계열 하드코딩은 예외로 보지 않는다.
gray는 항상 DS 토큰으로 처리한다.

## 5. 사용자에게 물어야 하는 경우

아래는 AI가 임의로 확정하지 말고 물어볼 수 있는 항목이다.

- 카드/컨테이너를 강조형으로 할지 플랫형으로 할지
- Drawer/특수 팝업의 노출 위치와 방향
- Gap 리스트로 할지 Gapless 리스트로 할지
- 상태색이 단순 장식인지 실제 의미 전달용인지 불명확한 경우

아래는 묻지 말고 기본값을 사용한다.
- 일반 텍스트 크기
- 일반 radius 값
- 일반 hover/transition 방식
- 툴팁 기본 placement/delay
- 기본 border 강도

## 6. 금지 패턴

- `bg-white`, `bg-black`, `bg-gray-*`
- `text-gray-*`
- `border-gray-*`
- `bg-[var(--app-*)]`
- `text-xs`, `text-sm`, `text-base`, `text-lg`
- `font-bold`, `font-light`, `font-extrabold`
- `transition-all`
- `rounded` 접미사 없는 모호한 radius 사용
- `rounded-[Npx]` 임의값 사용
- DS 컴포넌트를 감싸서 전혀 다른 컴포넌트처럼 만드는 방식
- 같은 리스트 안에서 gap과 border 구분 혼용

## 7. 빠른 체크리스트

- DS 컴포넌트로 대체 가능한가?
- 레이아웃 색상은 `app-*` 토큰을 썼는가?
- 일반 색상은 DS 시맨틱 토큰을 썼는가?
- 텍스트 크기를 `size-*`로 썼는가?
- 굵기를 3단계 안에서 썼는가?
- radius를 `sm/md/lg/full` 중 맥락에 맞게 썼는가?
- transition 속성을 명시했는가?
- 인터랙티브 요소 히트 영역이 충분한가?
- 툴팁이 필요한 요소에 적용했는가?
- 수정 범위를 벗어난 개선을 하지 않았는가?
