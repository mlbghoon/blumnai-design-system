# Deskit UI 제작 AI 규칙서

## 0. 목적 및 전제
1. **목표**: Human Designer의 70% 수준을 AI가 신속하게 제작하는 것을 목표로 한다.
2. **역할 분담**: AI는 가이드라인을 준수하여 일관성 있는 베이스 UI를 생성하며, 세부적인 미적 튜닝(Fine-tuning)은 사람이 직접 수행한다.
3. **근거**: 모든 작업은 Storybook에 정의된 컴포넌트와 디자인 시스템 변수(Design Tokens)를 최우선으로 한다. Storybook의 스토리와 TypeScript 인터페이스가 컴포넌트 이름, `variant`, `size`, `shape`의 단일 진실 원본이다.
4. **AI 컴포넌트 사용 원칙**: AI는 DS 컴포넌트를 코드로 작성하기 전에, 반드시 해당 컴포넌트의 TypeScript 인터페이스(Props)나 Storybook 파일을 조회하여 유효한 속성(`variant`, `size` 등)만 사용해야 한다. 임의의 속성 이름(환각)을 지어내지 않는다. Storybook과 문서 예시가 충돌하면 실제 Props 인터페이스를 우선한다.

[foundations/](foundations/README.md)에서 세부 디자인 요소를 참고한다.

---

## 0. 우선순위

항상 아래 순서로 판단한다.

1. DS 컴포넌트 사용 가능 여부
2. 색상 선택은 **영역 판단이 먼저**다.
   - 앱 레이아웃 영역(GNB, 사이드바, 대화방, 컴포저, 참조패널, 모달, 검색, 공지 등)이면 **앱 레이아웃 토큰**(`bg-app-*`, `text-app-*`, `border-app-*`) 우선
   - 그 외 일반 surface(카드, 패널, 콘텐츠, 리스트 등)는 **DS 시맨틱 토큰**(`bg-default`, `text-subtle`, `border-default` 등) 우선
   - 레이아웃 영역 안에 독립 surface가 들어오면, 바깥 껍데기는 `app-*`, 안쪽 독립 surface는 DS 시맨틱 토큰을 사용한다
3. 기존 파일/주변 코드 패턴 유지
4. 특수 케이스에 한해 하드코딩 허용 — **허용 목록은 §4 참조** (단일 진실 원본)

## 1. 절대 규칙

### 1.1 컴포넌트
- 버튼, 입력, 선택, 체크박스, 라디오, 스위치, 탭, 다이얼로그, 툴팁, 팝오버, 아바타, 뱃지, 스크롤 영역, 토스트, 날짜 선택, 데이터 그리드는 직접 구현하지 말고 DS 컴포넌트를 우선 사용한다.
- DS 컴포넌트가 있으면 자체 구현보다 DS를 사용한다.
- 정확한 컴포넌트 이름과 허용 prop은 Storybook 스토리와 TypeScript 인터페이스를 source of truth로 삼는다.
- DS 컴포넌트 스타일 변경은 공식 prop(`variant`, `size`, `shape`, `className`)으로만 한다.
- DS 내부 구조를 CSS 셀렉터로 강제 오버라이드하지 않는다.
- DS 컴포넌트를 감싸 완전히 다른 컴포넌트처럼 만들지 않는다.

### 1.2 색상
- `bg-white`, `bg-black`, `bg-gray-*` 사용 금지
- `text-gray-*` 사용 금지
- `border-gray-*` 사용 금지
- `bg-[var(--app-*)]` arbitrary value 사용 금지
- 레이아웃 색상은 `bg-app-*`, `text-app-*`, `border-app-*` 사용
- 일반 surface는 DS 배경 토큰 사용
- gray 계열 hex 하드코딩 금지

### 1.3 타이포그래피
- `text-xs`, `text-sm`, `text-base`, `text-lg` 등 Tailwind 텍스트 크기 사용 금지
- 텍스트 크기는 `size-*`만 사용
- `font-light`, `font-bold`, `font-extrabold` 사용 금지
- 굵기는 `font-normal`, `font-medium`, `font-semibold`만 사용

### 1.4 인터랙션
- `transition-all` 금지
- 변경되는 속성만 명시한 transition 사용
- 인터랙티브 요소는 최소 40x40px 히트 영역 확보
- 아이콘 전용 버튼에는 툴팁 적용

### 1.5 라운딩
- 임의 radius 값 금지
- 같은 역할의 요소는 같은 radius 사용
- 기본 3단계:
  - 말단 요소: `rounded-sm`
  - 컨테이너: `rounded-md`
  - 콘텐츠 블록/모달: `rounded-lg`

### 1.6 Primary 색상 사용
- Primary 색상은 시각적으로 매우 두드러지므로 남용 금지
- 버튼(`style="primary"`), 포인트 컬러, 강조 아이콘/뱃지 등 primary 계열 색상을 사용하는 모든 요소에 적용되는 규칙이다
- Primary는 기본값이 아니다. 화면에서 가장 중요한 **주 CTA 그룹**에만 우선 사용한다
- **1개 컨테이너(카드, 패널, 섹션 등)당 primary 색상 요소는 최대 1개**만 사용한다
- 리스트, 카드 그리드, 테이블 셀처럼 **같은 위계의 반복 구조에는 primary를 기본 적용하지 않는다**
- 같은 중요도의 액션이 여러 개 있으면 primary를 반복하지 말고 `style="secondary"` 또는 `style="soft"`를 사용한다
- primary를 배치할 때는 "이 화면에서 사용자의 시선이 가장 먼저 가야 하는 요소가 무엇인가"를 기준으로 판단한다
- 사용자가 반복 구조에 primary를 명시적으로 요청한 경우에는 따르되, 기본 규칙에서 벗어나는 요청임을 짧게 보고한다

## 2. UI 제작 결정 트리

### 2.1 새 UI를 만들 때
1. DS 컴포넌트로 표현 가능한지 확인한다.
2. 레이아웃 영역인지 일반 surface인지 판단한다.
3. 색상 토큰을 선택한다.
4. 텍스트 크기와 굵기를 선택한다.
5. spacing/radius/border/shadow를 선택한다.
6. 인터랙션과 툴팁 필요 여부를 판단한다.
7. z-index가 필요한 경우 기존 표 안에서 선택한다.

### 2.2 배경색 결정 순서
1. 앱 레이아웃 영역이면 `bg-app-*`
   - 단, 그 안에 들어가는 독립 카드/패널/폼 surface는 `bg-default` 또는 `bg-subtle`로 분리 가능
2. 일반 콘텐츠 배경이면 `bg-default`
3. 약한 구분이 필요하면 `bg-subtle`
4. 더 강한 구분, 토글 OFF, 핸들, placeholder 성격이면 `bg-muted`
5. 오버레이면 `bg-overlay`
6. 반전 강조면 `bg-inverted`
7. disabled 상태면 `bg-state-disabled`
8. 위 토큰으로 표현 불가한 특수 케이스 — **§4 참조**

### 2.3 텍스트 색상 결정 순서
1. 기본 본문: `text-default`
2. 보조 설명: `text-subtle`
3. 캡션/메타: `text-muted`
4. 힌트/placeholder: `text-hint`
5. 시맨틱 상태는 상태 토큰 또는 시맨틱 아이콘 토큰 사용

### 2.4 보더 결정 순서
1. 기본 구분선: `border-default`
2. 더 강한 구분: `border-darker`
3. 가장 강한 구분: `border-strong`
4. 상태 보더: `border-destructive`, `border-success`, `border-warning`, `border-informative`

### 2.5 radius 결정 순서
1. 버튼, 칩, 뱃지, 입력필드: `rounded-sm`
2. 이들을 감싸는 래퍼: `rounded-md`
3. 카드, 말풍선, 패널, 모달: `rounded-lg`
4. 원형/알약형만 `rounded-full`

### 2.6 카드/컨테이너 판단

**기본은 자동 판단**이다. 콘텐츠 성격을 보고 AI가 직접 결정하며, 사용자에게 묻지 않는다.

| 콘텐츠 성격 | 카드 유형 | 적용 규칙 |
| --- | --- | --- |
| 클릭 유도, 핵심 요약, 독립적으로 돋보여야 하는 블록 | **강조 카드** | `rounded-lg` 적용 후 `border-default` 또는 `shadow-card` 선택 |
| 정보 나열, 설정 패널, 반복 목록, 연속 콘텐츠 영역 | **플랫 카드** | shadow 없이 `bg-default` 또는 `bg-subtle` + padding 중심 구성 |

추가 판단:
- 대시보드 KPI/요약 위젯처럼 **같은 형태가 2개 이상 반복**되면 기본은 플랫 카드다. 특정 카드 1개만 더 중요할 때만 강조 카드를 사용한다.
- Empty state, hero의 단일 요약 블록처럼 **단독으로 화면 의미를 대표**하면 강조 카드 우선이다.
- 사이드바/참조패널 안의 필터 박스처럼 **레이아웃 안의 보조 설정 블록**은 기본 플랫 카드다.

**둘 다 자연스러운 경우에만** 사용자에게 강조/플랫 중 어느 쪽인지 묻는다. 기준은 아래와 같다.
- 강조와 플랫이 모두 정보 위계를 크게 해치지 않을 때
- 한쪽을 선택해도 CTA 강도, 읽기 흐름, 레이아웃 밀도가 명확히 나빠지지 않을 때

radius/shadow 수치를 직접 묻지 않는다.

### 2.7 중첩 컨테이너 규칙

컨테이너 안에 컨테이너를 넣는 경우 아래 규칙을 따른다.

1. **안쪽 R값 < 바깥 R값** — 내부 요소가 외부보다 큰 radius를 가지면 안 된다 (상세: [border-radius.md](foundations/border-radius.md))
2. **바깥이 안쪽보다 같거나 강한 구분 스타일을 가진다** — 내부 컨테이너가 외부보다 시각적으로 튀면 위계가 역전된다
3. **중첩 시 기본 구분은 border** — shadow가 가장 강한 구분이지만, 중첩에서 무조건 shadow를 쓸 필요는 없다. 중첩 구분이 필요하면 `border-default`를 우선 사용한다

구분 강도 순서: `없음` < `bg 차이` < `border` < `shadow`

```text
✅ 바깥 shadow-card, 안쪽 border-default   → 바깥이 더 강함
✅ 바깥 border-default, 안쪽 bg-subtle      → 바깥이 더 강함
✅ 바깥 border-default, 안쪽 border-default → 동일 (허용)
❌ 바깥 bg-subtle, 안쪽 shadow-card         → 안쪽이 더 강함 (역전)
```

### 2.8 리스트 판단
1. 표 형태 데이터면 DS Table/DataGrid 사용
2. 자유 레이아웃 나열이면 Card List
3. Card List는 아래 둘 중 하나만 선택
- Gap 리스트: 아이템 사이 간격으로 구분
- Gapless 리스트: 컨테이너 내부 구분선으로 구분

금지:
- 같은 리스트에서 gap과 `border-b`를 혼용하지 않는다.

## 3. 영역별 규칙

### 3.1 레이아웃
- 전체 컨테이너: `bg-app-container`
- GNB: `bg-app-gnb`, `text-app-gnb-text`, `border-app-gnb-border`
- 사이드바: `bg-app-sidebar`
- 대화방 헤더/본문/컴포저: 각 `bg-app-room-*`, `bg-app-composer`
- 참조 패널: `bg-app-ref-*`
- 모달: `bg-app-modal`
- 검색/공지: `bg-app-search`, `bg-app-notice`

### 3.2 타이포그래피
- 캡션/뱃지/메타: `size-xs`
- 본문/라벨/리스트 아이템: `size-sm`
- 섹션 제목/강조: `size-md`
- 페이지/패널 제목: `size-lg`

### 3.3 폰트 굵기
- 본문: `font-normal`
- 버튼/입력 라벨: `font-medium`
- 섹션 제목/강조: `font-semibold`

### 3.4 간격
- DS spacing은 숫자가 곧 px 값이다. 같은 숫자의 DS 유틸리티가 있으면 그대로 사용한다. 예: `p-4` → `padding-4`, `px-8` → `padding-x-8`
- `p-*`, `px-*`, `py-*`는 DS `padding-*` 계열로 변환한다 (DS에 해당 값이 있는 경우).
- `gap-*`은 DS `ds-gap-*`로 변환한다. Tailwind `gap-*` 사용 금지.
- `margin`, `space-*`는 DS 지원 범위가 제한적이므로, 값이 있으면 DS 우선, 없으면 Tailwind 유지.
- DS에 같은 숫자 토큰이 없으면 가장 가까운 값으로 억지 변환하지 말고 Tailwind 값을 유지한다.
- `panelSpacing.ts` 상수가 있으면 우선 사용한다.

### 3.5 인터랙션
- 리스트 아이템: `hover:bg-subtle transition-colors`
- ghost 버튼: `hover:bg-muted transition-colors`
- 아이콘 버튼: `hover:bg-muted transition-colors`
- 카드 hover: `hover:shadow-card transition-shadow` (플랫 카드가 hover 시 미세하게 들어올리는 경우에만 적용. 이미 `shadow-card`인 강조 카드는 hover 변화 없음)
- press feedback: 필요 시 `active:scale-[0.96]`

### 3.6 모달
- 세로 크기는 기본적으로 콘텐츠 높이에 맞춘다
- 뷰포트를 넘기면 `max-h-[calc(100vh-48px)]` + 내부 스크롤
- 특별한 지시가 없으면 중앙 dissolve 방식으로 노출

### 3.7 툴팁
- HTML `title` 사용 금지
- `TooltipTrigger` 사용
- 아이콘 전용 버튼: 툴팁 필수
- 잘린 텍스트: 툴팁 필수
- 단축키 있는 액션: `badge` 사용 가능

## 4. 허용되는 예외 (단일 진실 원본)

> 하드코딩 허용 목록은 이 섹션이 유일한 출처다. §0과 §2.2는 이 섹션을 참조한다.

DS 토큰으로 표현할 수 없는 경우에만 허용한다.

- 상태 플래그 색상 맵
- 브랜드/아이덴티티 컬러
- 채팅 말풍선별 개별 색상
- gradient 배경
- 시맨틱 토큰에 없는 유색 아이콘/배경

예외 적용 전 자가 점검:
- Storybook/토큰 목록에 같은 의미의 시맨틱 색이 이미 있는가?
- 이 색이 상태 표현인가, 브랜드/데이터/장식/특수 구분인가?
- gray 계열을 유채색 예외로 잘못 처리하고 있지 않은가?

**제외**: 회색 계열 하드코딩은 예외로 보지 않는다. gray는 항상 DS 토큰으로 처리한다.

## 5. 사용자에게 물어야 하는 경우

아래는 AI가 임의로 확정하지 말고 물어볼 수 있는 항목이다. **기본은 자동 판단**이고, 자동 판단이 둘 다 자연스러운 경우에만 묻는다.

- 카드/컨테이너 유형 — 콘텐츠 성격이 강조와 플랫 양쪽으로 모두 자연스러울 때만 (§2.6 참조)
- Drawer/특수 팝업의 노출 위치와 방향
- Gap/Gapless 리스트 — 콘텐츠 밀도와 구분 의도가 양쪽으로 모두 자연스러울 때만
- 상태색이 단순 장식인지 실제 의미 전달용인지 불명확한 경우

아래는 묻지 말고 기본값을 사용한다.
- 일반 텍스트 크기
- 일반 radius 값
- 일반 hover/transition 방식
- 툴팁 기본 placement/delay
- 기본 border 강도

- `text-gray-*`
- `border-gray-*`
- `bg-[var(--app-*)]`
- `text-xs`, `text-sm`, `text-base`, `text-lg`
- `font-bold`, `font-light`, `font-extrabold`
- `transition-all`
- `rounded` 접미사 없는 모호한 radius 사용
- `rounded-[Npx]` 임의값 사용
- DS 컴포넌트를 감싸서 전혀 다른 컴포넌트처럼 만드는 방식
- 같은 리스트 안에서 gap과 border 구분 혼용

---

## 7. 빠른 체크리스트

- DS 컴포넌트로 대체 가능한가?
- 레이아웃 색상은 `app-*` 토큰을 썼는가?
- 일반 색상은 DS 시맨틱 토큰을 썼는가?
- 텍스트 크기를 `size-*`로 썼는가?
- 굵기를 3단계 안에서 썼는가?
- radius를 `sm/md/lg/full` 중 맥락에 맞게 썼는가?
- transition 속성을 명시했는가?
- 인터랙티브 요소 히트 영역이 충분한가?
- 툴팁이 필요한 요소에 적용했는가?
- 수정 범위를 벗어난 개선을 하지 않았는가?
