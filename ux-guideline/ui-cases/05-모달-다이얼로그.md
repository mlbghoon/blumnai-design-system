# 케이스 05: 모달/다이얼로그

## 화면 구조

### 유형 A: 확인 다이얼로그

```
┌─────────────────────────────────┐
│ 항목 삭제                    ✕  │
├─────────────────────────────────┤
│                                 │
│ 이 항목을 삭제하시겠습니까?       │
│ 삭제된 항목은 복구할 수 없습니다. │
│                                 │
│              [취소]  [삭제]      │  ← secondary, destructive
└─────────────────────────────────┘
```

### 유형 B: 폼 다이얼로그

```
┌─────────────────────────────────┐
│ 새 항목 등록                 ✕  │
├─────────────────────────────────┤
│                                 │
│ 이름    [___________________]   │
│ 설명    [___________________]   │
│ 카테고리 [▼ 선택하세요     ]    │
│                                 │
│              [취소]  [등록]      │  ← secondary, primary
└─────────────────────────────────┘
```

### 유형 C: 정보 표시 다이얼로그

```
┌─────────────────────────────────┐
│ 상세 정보                    ✕  │
├─────────────────────────────────┤
│                                 │
│ 이름: 홍길동                     │
│ 이메일: hong@example.com        │
│ 등록일: 2026-03-15              │
│                                 │
│                        [닫기]   │  ← secondary
└─────────────────────────────────┘
```

## Primary 배치 판단

| 유형 | 확인 버튼 | 스타일 | 이유 |
|------|-----------|--------|------|
| 삭제/위험 확인 | [삭제] | `destructive` | 위험 액션은 primary가 아닌 destructive |
| 생성/등록 | [등록] | `primary` | 긍정적 핵심 액션 |
| 수정/저장 | [저장] | `primary` | 긍정적 핵심 액션 |
| 정보 확인 | [닫기] | `secondary` | 단순 닫기에 primary 불필요 |
| 취소 | [취소] | `secondary` | 항상 보조 액션 |

- 모달은 그 자체가 1개 컨테이너 → primary 최대 1개
- 전체 화면 primary 합계: 모달 내 **1개** (+ 뒷배경 화면의 primary는 딤 처리로 비활성)

## 버튼 배치 규칙

- 우측 정렬: 확인 버튼이 오른쪽, 취소가 왼쪽
- 순서: `[취소(secondary)]  [확인(primary 또는 destructive)]`
- 닫기(✕) 아이콘: 우측 상단, `ghost` 스타일

## 흔한 실수

- 삭제 확인에서 [삭제]를 primary로 만드는 것 → `destructive` 사용
- [닫기]를 primary로 만드는 것 → 정보 표시용 모달의 닫기는 `secondary`
- 모달 내부에 primary 버튼 + primary 색상 링크를 동시에 넣는 것 → 컨테이너당 1개 규칙
- [취소]와 [확인] 모두 `ghost`로 만들어 위계가 불명확한 것 → 확인은 반드시 취소보다 시각적으로 강해야 한다
