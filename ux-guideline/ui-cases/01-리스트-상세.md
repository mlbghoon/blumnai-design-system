# 케이스 01: 리스트 + 상세 화면

## 화면 구조

```
┌─────────────────────────────────────────────────┐
│ 헤더                                [+ 새 항목] │  ← primary 후보 ①
├──────────────┬──────────────────────────────────┤
│ 리스트       │ 상세 패널                         │
│              │                                  │
│ ● 항목 A     │  제목: 항목 A                     │
│   항목 B     │  내용: ...                        │
│   항목 C     │                                  │
│   항목 D     │  [수정]  [삭제]                   │  ← secondary, ghost
│              │                          [저장]  │  ← primary 후보 ②
└──────────────┴──────────────────────────────────┘
```

## Primary 배치 판단

| 위치 | 역할 | 스타일 | 이유 |
|------|------|--------|------|
| [+ 새 항목] | 새 항목 생성 | `primary` | 이 화면의 핵심 CTA |
| [저장] | 변경 저장 | `primary` | 상세 패널의 핵심 CTA |
| [수정] | 편집 모드 전환 | `secondary` | 보조 액션 |
| [삭제] | 항목 삭제 | `ghost` + destructive | 위험 액션, 시각적 약화 |

- 헤더 영역(컨테이너 ①)에 primary 1개: [+ 새 항목]
- 상세 패널(컨테이너 ②)에 primary 1개: [저장]
- 전체 화면 primary 합계: **2개**

## 리스트 구분 방식

둘 중 하나만 선택한다 (혼용 금지):

- **Gap 리스트**: 아이템 간 간격으로 시각 구분 → 카드형 아이템일 때
- **Gapless 리스트**: 컨테이너 내부 `border-b`로 구분 → 텍스트 중심 아이템일 때

## 흔한 실수

- 리스트 각 아이템에 primary 버튼을 넣는 것 → 리스트 아이템 내 액션은 `ghost` 또는 아이콘 버튼
- [수정]과 [저장]을 모두 primary로 만드는 것 → 같은 컨테이너에 primary 2개 금지
- 리스트 필터/정렬 버튼을 primary로 만드는 것 → 유틸리티 액션은 `ghost`
