## 1. 시맨틱 soft 배경 토큰 비대칭

**현황**
- `bg-state-destructive-soft`만 존재
- `bg-state-informative-soft`, `bg-state-warning-soft`, `bg-state-success-soft`는 정의되어 있지 않음
- `border-{informative|warning|success|destructive}` 와 `text-{informative|warning|success|destructive}` 는 4종 모두 존재 → border/text는 대칭, 배경만 비대칭

**문제**
- 정보·경고·성공 톤의 콜아웃을 만들 때 시맨틱 soft 배경 패턴을 쓸 수 없다
- DS `InfoBox` 컴포넌트가 있어 우회는 가능하지만, 인라인 배지/태그/하이라이트 등 InfoBox로 감싸기 어색한 케이스에서는 대안이 없다

**참고 위치**
- 토큰 정의: [src/styles/tokens/colors.css](../src/styles/tokens/colors.css)
- 유틸리티: [src/styles/utilities.css](../src/styles/utilities.css)

**제안**
- `--bg-state-informative-soft`, `--bg-state-warning-soft`, `--bg-state-success-soft` 토큰 추가
- 4개 테마(Light-A, Dark, Theme-B Light, Theme-B Dark) 모두에 대응

## 5. `shadow-card` 토큰의 배경 의존성

**현황**
- `--shadow-card: #00000006 0px 1px 1px, inset 0px -1px 0px #00000026`
- 8자리 hex에서 outer shadow alpha = `0x06` = **2.35%**, inset bottom line alpha = `0x26` = **14.9%**
- `bg-default`(흰/near-white) 위에서 `border-default`와 쌍으로 쓸 때만 "미세한 입체감"으로 의미가 생기는 값

**문제**
- 유색 배경(slate, blue, mobile chat bg 등) 위에 카드를 얹으면 outer shadow 2.35%가 사실상 보이지 않아 elevation이 표현되지 않음 — "배경 위에 떠 있는 카드" 감각이 사라짐
- 이름(`shadow-card`)만 보고 "카드 컨테이너용 범용 그림자"로 오해하기 쉬움
- `shadow-modal-sm/md/lg`는 멀티레이어 블러 캐스케이드(`0x0a ≈ 4%`)로 유색 배경에서도 자연스럽게 스머지됨 — elevation이 필요한 컨텍스트에선 `shadow-modal-sm`이 실질 대안

**참고 위치**
- 토큰 정의: `dist/index.css` 내 `--shadow-card`, `--shadow-modal-sm` 변수
- UX Guideline 원문: `ux-guideline/foundations/elevation.md`, `components.md` §6 (현재 "emphasized = shadow-card + border-default + rounded-lg"로 기술)

**제안**
- (단기) 가이드라인 문서에 **"shadow-card는 흰 배경 전용"** 명시 + 유색 배경/플로팅 컨텍스트에는 `shadow-modal-sm` 권장
- (중기) 용법 분리를 위한 신규 토큰 검토
  - 예: `shadow-bubble` — 채팅 말풍선/모바일 카드용 중간 강도 elevation
  - 또는 `shadow-card`의 outer alpha를 `0x0a` 수준으로 상향 (기존 사용처 영향 면적 스캔 필요)

## 6. Tailwind v4 유틸명과 DS 컬러 토큰명 네이밍 컬리전

**현황**
- `dist/index.css`에서 `.shadow-card` 셀렉터가 3벌 생성됨:
  1. `.shadow-card { box-shadow: var(--shadow-card) }` — 의도한 컴파운드 섀도우
  2. `.shadow-card { --tw-shadow-color: var(--card) }` — Tailwind v4가 `card`를 "컬러명"으로 오해해서 생성한 shadow-color modifier 규칙
  3. `.shadow-card { --tw-shadow-color: color-mix(in oklab, var(--card) var(--tw-shadow-alpha), transparent) }` — @supports fallback
- Tailwind v4의 `shadow-{colorName}` 패턴이 DS 토큰명 `--card`(카드 배경색)를 컬러로 인식한 결과

**문제**
- 기능상 버그는 없음(1번 규칙이 cascade로 이김)이지만 CSS 내 **불필요한 규칙 중복** 발생
- 더 중요한 점: 같은 패턴이 다른 토큰에도 재발할 가능성
  - `bg-card`, `text-card`, `border-card`, `ring-card`, `outline-card` 등
  - `bg-subtle`, `bg-muted`, `bg-default` 등 컬러 토큰명이 Tailwind 컬러 네임스페이스와 만나는 모든 지점
- v4 이전엔 컬러 팔레트가 config 기반이어서 충돌이 제한적이었으나, v4는 이름 기반 추론이 강해 부산물이 늘어남

**참고 위치**
- 생성물 확인: `dist/index.css`에서 `grep -c "shadow-card\|bg-card\|text-card"` 셀렉터 중복 확인
- 원인: `@theme` 내 `--card: ...`(컬러) 선언이 Tailwind v4의 자동 유틸 생성 트리거가 됨

**제안**
- DS 내 모든 컬러 토큰명 전수조사 후 Tailwind 자동 유틸 충돌 지점 리스트업
  - `shadow-*`, `ring-*`, `outline-*`, `divide-*`, `accent-*`, `caret-*` 네임스페이스 전부 점검
- 장기적으로 토큰명 리브랜딩 고려 — 예: `--card` → `--surface-card`, `--shadow-card` → `--elevation-card`
- 또는 `@theme` 내에서 `--tw-*` 자동 매핑을 명시적으로 opt-out하는 방법 탐색

## 7. 토큰 네이밍이 사용 컨텍스트 제약을 담지 못함

**현황**
- `shadow-card` 예시처럼, 이름만으로는 "어떤 배경 위에서 쓰는 토큰인지" 제약이 안 읽힘
- 가이드라인에 `components.md` §6에 "Emphasized = shadow-card + border-default + rounded-lg" 공식만 있고, **어떤 배경 전제 하에** 그 공식이 유효한지는 암묵

**문제**
- 개발자/디자이너가 DS 규칙을 "범용 공식"으로 받아들여 배경 변경 시 시각 품질 저하를 자각하지 못함
- 특히 AI 보조 생성(Figma Make, Claude 등) 시 컨텍스트 제약이 프롬프트에 안 담기므로 오적용 빈도가 높아질 가능성

**제안**
- 토큰 문서(Storybook 또는 foundations)에 **배경 compatibility matrix** 도입
  - 가로축: 배경 토큰(`bg-default` / `bg-subtle` / `bg-muted` / 유색)
  - 세로축: 각 elevation/shadow/border 토큰
  - 셀: ✓ 권장 / ⚠ 시각 약화 / ✗ 사실상 무효
- 또는 토큰 명명 규칙에 컨텍스트 힌트 접미사 도입 검토 (예: `shadow-card-onSurface`, `shadow-bubble-onCanvas`)
- 최소한 `components.md` §6 공식에 배경 전제 한 줄 추가
